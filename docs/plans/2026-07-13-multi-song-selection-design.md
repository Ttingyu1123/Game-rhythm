# 多歌曲選曲系統設計

## 目標

把目前單曲流程改成資料驅動的曲庫。現階段只保留已完成的〈月光齒輪巡遊〉作為可玩歌曲，並顯示一個停用的新曲槽位；不以測試音樂冒充第二首正式歌曲。

## 核心選擇

每首歌曲放在獨立模組，歌曲目錄只負責登錄與查找。歌曲模組同時提供顯示資料、音訊來源、難度清單與譜面工廠，因此 `Game` 不再知道特定歌曲的檔名或常數名稱。

沒有採用瀏覽器自動掃描資料夾。MP3 無法可靠提供 BPM、首拍偏移、難度與譜面；明確登錄歌曲能在開局前驗證資料，也能用穩定的歌曲 ID 分開保存紀錄。

## 資料結構

- `src/data/songs/moonlitGearParade.js`：〈月光齒輪巡遊〉的完整歌曲定義。
- `src/data/songCatalog.js`：建立不可變曲庫、驗證歌曲 ID、依 ID 查找歌曲。
- `src/data/songs.js`：保留相容匯出，避免既有測試與外部引用中斷。
- 每首歌曲包含 `id`、中英文曲名、作者、章節、BPM、調性、長度、音訊、難度與 `createChart()`。

正式音訊網址留在歌曲模組中，使用靜態 `new URL(..., import.meta.url)`，讓 Vite 在建置時複製並雜湊 MP3。

## 切歌流程

選曲只允許在選單狀態執行。選中歌曲後，遊戲會：

1. 停止舊音源並清除已解碼的舊歌曲音訊。
2. 更新 AudioEngine 的長度、BPM 與載入函式。
3. 切到該歌曲的預設難度。
4. 更新歌曲卡、遊戲 HUD、結算標題與最佳紀錄。

AudioContext、音量節點與使用者音量設定會保留，避免每次切歌重建整套音訊圖。

## 選曲介面

右側面板最上方新增「星圖曲庫」。每首可玩歌曲是一顆原生按鈕，選中狀態同時使用外觀與 `aria-pressed` 表達。列表下方放置停用的新曲槽位，清楚標示「等待新旋律」，不讓玩家誤以為它能點擊。

主歌曲卡仍是詳細資訊區，會隨選曲更新曲名、章節、BPM、調性、長度、來源與難度。桌機顯示在右側欄；980px 以下曲庫跨兩欄；760px 以下改為單欄。曲庫可垂直捲動，因此日後增加歌曲不會讓整頁無限增高。

## 擴充新歌

加入新歌只需：

1. 放入 MP3。
2. 複製歌曲模組並填寫 metadata、音訊 URL、難度與譜面。
3. 在 `SONG_CATALOG` 加入該歌曲。

最佳紀錄沿用既有 `songId + difficultyId` 儲存方式，不需另做資料遷移。

## 驗證範圍

- 單元測試：曲庫驗證、查找與重複 ID；音訊切換會停止舊音源、清除舊 buffer 並套用新時間軸。
- 桌機 E2E：曲庫數量、選中狀態、空槽、難度、開局與紀錄。
- 手機 E2E：390px 無水平溢出，選曲按鈕與空槽完整可見。
- 正式建置：MP3 仍由 Vite 打包，無 JavaScript 或資產錯誤。
